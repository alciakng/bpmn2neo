# Neo4j Graph Schema Documentation

## Overview

This document describes the Neo4j graph database schema generated by `bpmn2neo` when converting BPMN 2.0 XML files to a property graph model. The schema is designed to preserve all BPMN semantics while enabling powerful graph-based queries and analysis.

---

## Node Labels

### 1. Container Nodes

#### `Category`
Represents a hierarchical category for organizing BPMN models (e.g., business domains, departments, process groups).

**Properties:**
- `id` (string, required): Category node ID in format `{modelKey}_category`
- `modelKey` (string, required): Category identifier (same as name)
- `name` (string, required): Display name of the category
- `containerType` (string): Type of the containing entity
- `containerId` (string): ID of the containing entity

**Usage:**
- Categories form a hierarchical tree structure for organizing BPMN models
- Top-level categories connect to container nodes via `HAS_CATEGORY`
- Subcategories connect to parent categories via `HAS_SUBCATEGORY`
- BPMN models belong to categories via `CONTAINS_MODEL`

**Example:**
```
Container → HAS_CATEGORY → Category(HR) → HAS_SUBCATEGORY → Category(Recruitment)
                                                              → CONTAINS_MODEL → BPMNModel(Interview Process)
```

#### `BPMNModel`
Represents a BPMN collaboration or model (top-level container for processes and participants).

**Properties:**
- `id` (string, required): Unique identifier from BPMN XML
- `modelKey` (string, required): Model version/instance key
- `name` (string): Display name of the model
- `containerType` (string): Type of the containing entity (e.g., "Project", "Question")
- `containerId` (string): ID of the containing entity
- Additional attributes from `<collaboration>` element

**Constraints:**
```cypher
CREATE CONSTRAINT bpmn_model_nodekey IF NOT EXISTS
FOR (n:BPMNModel) REQUIRE (n.id, n.modelKey) IS NODE KEY
```

### 2. Flow Node Labels

#### `Activity`
Represents all types of BPMN activities (tasks, subprocesses, call activities, etc.).

**Properties:**
- `id` (string, required): Unique identifier
- `modelKey` (string, required): Model version/instance key
- `name` (string): Activity name
- `activityType` (string, required): Type of activity
  - `"Task"`: Generic task
  - `"User"`: User task
  - `"Service"`: Service task
  - `"Send"`: Send task
  - `"Receive"`: Receive task
  - `"Manual"`: Manual task
  - `"BusinessRule"`: Business rule task
  - `"Script"`: Script task
  - `"Call"`: Call activity
  - `"SubProcess"`: Subprocess
  - `"Transaction"`: Transaction subprocess
- `containerType` (string): Container type
- `containerId` (string): Container ID
- `isForCompensation` (boolean, optional): Whether activity is for compensation
- `default` (string, optional): Default outgoing flow ID
- `startQuantity` (integer, optional, default: 1): Start quantity
- `completionQuantity` (integer, optional, default: 1): Completion quantity

**Activity Type Specific Properties:**
- **User/Service/BusinessRule Tasks:**
  - `implementation` (string): Implementation type
- **Script Task:**
  - `scriptFormat` (string): Script language format
- **Send/Receive Tasks:**
  - `messageRef` (string): Associated message reference
  - `operationRef` (string): Associated operation reference
- **Receive Task:**
  - `instantiate` (boolean): Whether it instantiates the process
- **Call Activity:**
  - `calledElement` (string): Called process/subprocess ID
- **SubProcess:**
  - `triggeredByEvent` (boolean): Whether it's an event subprocess
- **Transaction:**
  - `method` (string, default: "##Compensate"): Transaction method

**Loop Characteristics:**
- `hasStandardLoop` (boolean): Has standard loop characteristics
- `standardLoopTestBefore` (boolean): Test condition before execution
- `standardLoopMaximum` (integer): Maximum loop iterations
- `hasMultiInstance` (boolean): Has multi-instance characteristics
- `multiInstanceSequential` (boolean): Sequential multi-instance
- `multiInstanceBehavior` (string): Behavior ("All", "One", "Complex")

**Constraints:**
```cypher
CREATE CONSTRAINT activity_nodekey IF NOT EXISTS
FOR (n:Activity) REQUIRE (n.id, n.modelKey) IS NODE KEY
```

## Relationship Types

### 1. Structural Hierarchy

#### `HAS_CATEGORY`
Connects a container to its top-level category nodes.

**Pattern:**
```cypher
(Container)-[:HAS_CATEGORY]->(Category)
```

**Properties:**
- `modelKey` (string): Category model key
- `containerType` (string): Container type
- `containerId` (string): Container ID

**Usage:**
- Links containers to their root-level categories
- Created when a category is defined without a parent
- Enables category tree navigation from container root

**Example:**
```cypher
// Create top-level category under container
MERGE (c {id: 'projectA'})
MERGE (cat:Category {id: 'Finance_category', modelKey: 'Finance', name: 'Finance'})
CREATE (c)-[:HAS_CATEGORY {containerType: 'Project', containerId: 'projectA', modelKey: 'Finance'}]->(cat)
```

---

#### `HAS_SUBCATEGORY`
Connects a parent category to its child subcategories.

**Pattern:**
```cypher
(Category:Parent)-[:HAS_SUBCATEGORY]->(Category:Child)
```

**Properties:**
- `modelKey` (string): Child category model key
- `containerType` (string): Container type
- `containerId` (string): Container ID

**Usage:**
- Creates hierarchical category structures (tree)
- Enables drill-down navigation: Category → Subcategory → BPMN Models
- Supports multi-level organization (e.g., Department → Team → Project)

**Example:**
```cypher
// Create subcategory hierarchy
MATCH (parent:Category {modelKey: 'Finance'})
MERGE (child:Category {id: 'Accounting_category', modelKey: 'Accounting', name: 'Accounting'})
CREATE (parent)-[:HAS_SUBCATEGORY {containerType: 'Project', containerId: 'projectA', modelKey: 'Accounting'}]->(child)
```

---

#### `CONTAINS_MODEL`
Connects a category to the BPMN models it organizes.

**Pattern:**
```cypher
(Category)-[:CONTAINS_MODEL]->(BPMNModel)
```

**Properties:**
- `modelKey` (string): BPMN model key
- `containerType` (string): Container type
- `containerId` (string): Container ID

**Usage:**
- Associates BPMN models with their organizational category
- Created during BPMN loading when `parent_category_key` is specified
- Enables category-based model filtering and search

**Example:**
```cypher
// Link BPMN model to category
MATCH (cat:Category {modelKey: 'Accounting'})
MATCH (model:BPMNModel {modelKey: 'InvoiceApproval'})
CREATE (cat)-[:CONTAINS_MODEL {containerType: 'Project', containerId: 'projectA', modelKey: 'InvoiceApproval'}]->(model)
```

---

#### `NEXT_PROCESS`
Defines sequential ordering between BPMN models (process flow continuation).

**Pattern:**
```cypher
(BPMNModel:Predecessor)-[:NEXT_PROCESS]->(BPMNModel:Successor)
```

**Properties:**
- `modelKey` (string): Successor model key
- `containerType` (string): Container type
- `containerId` (string): Container ID

**Usage:**
- Chains BPMN models in execution or logical sequence
- Created when `predecessor_model_key` is provided during loading
- Enables process mining analysis across model boundaries
- Supports 4-hop predecessor/successor queries for context understanding

**Example:**
```cypher
// Create process sequence
MATCH (prev:BPMNModel {modelKey: 'OrderReceiving'})
MATCH (next:BPMNModel {modelKey: 'OrderFulfillment'})
CREATE (prev)-[:NEXT_PROCESS {containerType: 'Project', containerId: 'projectA', modelKey: 'OrderFulfillment'}]->(next)
```

**Analysis Use Cases:**
- Trace end-to-end process chains: `MATCH path = (start:BPMNModel)-[:NEXT_PROCESS*]->(end:BPMNModel)`
- Find predecessors: `MATCH (m:BPMNModel)<-[:NEXT_PROCESS*1..4]-(pred)`
- Find successors: `MATCH (m:BPMNModel)-[:NEXT_PROCESS*1..4]->(succ)`
- Build model flow diagrams for LLM context composition

#### `OWNS_NODE`
Defines ownership of flow nodes by process or lane.

**Pattern:**
```cypher
(Process|Lane)-[:OWNS_NODE]->(Activity|Event|Gateway)
```

**Properties:**
- `modelKey` (string): Model version/instance key
- `containerType` (string): Container type
- `containerId` (string): Container ID

**Logic:**
- If a process has lanes: Flow nodes are owned by their respective lanes
- If a process has no lanes: All flow nodes are owned by the process
- Unassigned nodes in a lane-based process: Owned by the process

- `containerType` (string): Container type
- `containerId` (string): Container ID

**Note:** Used for SubProcess, Transaction, and AdHocSubProcess activities.

## Common Property Patterns

All nodes and relationships include these common properties for multi-tenancy and versioning:

- `containerType` (string): Type of the container (e.g., "Project", "Question")
- `containerId` (string): Unique identifier of the container
- `modelKey` (string): Version/instance key for the BPMN model

These properties enable:
1. **Multi-tenancy**: Isolate data by container
2. **Versioning**: Track different versions of the same model
3. **Cleanup**: Efficiently delete all data for a specific model/container

## Schema Constraints

The schema uses composite node key constraints to ensure uniqueness and improve query performance:

```cypher
// Node key constraints (id + modelKey)
CREATE CONSTRAINT category_nodekey IF NOT EXISTS
FOR (n:Category) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT bpmn_model_nodekey IF NOT EXISTS
FOR (n:BPMNModel) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT participant_nodekey IF NOT EXISTS
FOR (n:Participant) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT process_nodekey IF NOT EXISTS
FOR (n:Process) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT lane_nodekey IF NOT EXISTS
FOR (n:Lane) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT activity_nodekey IF NOT EXISTS
FOR (n:Activity) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT event_nodekey IF NOT EXISTS
FOR (n:Event) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT gateway_nodekey IF NOT EXISTS
FOR (n:Gateway) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT data_nodekey IF NOT EXISTS
FOR (n:Data) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT dataref_nodekey IF NOT EXISTS
FOR (n:DataReference) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT group_nodekey IF NOT EXISTS
FOR (n:GROUP) REQUIRE (n.id, n.modelKey) IS NODE KEY;

CREATE CONSTRAINT textanno_nodekey IF NOT EXISTS

```

**Benefits:**
- Ensures uniqueness of (id, modelKey) pairs
- Automatically creates indexes for faster lookups
- Enables efficient MERGE operations during loading

## Example Queries

### Category Navigation

#### List all top-level categories
```cypher
MATCH (c {id: 'projectA'})-[:HAS_CATEGORY]->(cat:Category)
RETURN cat.name, cat.modelKey
ORDER BY cat.name
```

#### Navigate category tree hierarchy
```cypher
MATCH path = (root:Category)-[:HAS_SUBCATEGORY*]->(leaf:Category)
WHERE root.modelKey = 'Finance'
RETURN path
```

#### Find all models in a category (including subcategories)
```cypher
MATCH (cat:Category {modelKey: 'Finance'})-[:HAS_SUBCATEGORY*0..]->(subcat:Category)
MATCH (subcat)-[:CONTAINS_MODEL]->(model:BPMNModel)
RETURN cat.name as category, subcat.name as subcategory, model.name as model_name
```

#### Build category tree with model counts
```cypher
MATCH (cat:Category)
OPTIONAL MATCH (cat)-[:HAS_SUBCATEGORY*0..]->(subcat:Category)-[:CONTAINS_MODEL]->(m:BPMNModel)
WITH cat, count(DISTINCT m) as model_count
RETURN cat.name, cat.modelKey, model_count
ORDER BY cat.name
```

### Process Sequencing

#### Find immediate predecessor and successor models
```cypher
MATCH (model:BPMNModel {modelKey: 'OrderProcessing'})
OPTIONAL MATCH (pred:BPMNModel)-[:NEXT_PROCESS]->(model)
OPTIONAL MATCH (model)-[:NEXT_PROCESS]->(succ:BPMNModel)
RETURN pred.name as predecessor, model.name as current, succ.name as successor
```

#### Trace 4-hop process chain (backward)
```cypher
MATCH path = (model:BPMNModel {modelKey: 'OrderDelivery'})<-[:NEXT_PROCESS*1..4]-(predecessors:BPMNModel)
RETURN predecessors.name, length(path) as hops
ORDER BY hops DESC
```

#### Trace 4-hop process chain (forward)
```cypher
MATCH path = (model:BPMNModel {modelKey: 'OrderReceiving'})-[:NEXT_PROCESS*1..4]->(successors:BPMNModel)
RETURN successors.name, length(path) as hops
ORDER BY hops
```

#### Find complete end-to-end process flow
```cypher
// Find all models with no predecessors (entry points)
MATCH (start:BPMNModel)
WHERE NOT EXISTS {(start)<-[:NEXT_PROCESS]-()}
// Traverse to all successors
MATCH path = (start)-[:NEXT_PROCESS*]->(end:BPMNModel)
RETURN start.name as entry_point, [node in nodes(path) | node.name] as flow_sequence, length(path) as chain_length
ORDER BY chain_length DESC
```

#### Build model flow context for LLM (4-hop window)
```cypher
MATCH (model:BPMNModel {modelKey: 'InvoiceApproval'})
OPTIONAL MATCH pred_path = (model)<-[:NEXT_PROCESS*1..4]-(predecessors:BPMNModel)
OPTIONAL MATCH succ_path = (model)-[:NEXT_PROCESS*1..4]->(successors:BPMNModel)
WITH model,
     collect(DISTINCT {name: predecessors.name, modelKey: predecessors.modelKey, hops: length(pred_path)}) as preds,
     collect(DISTINCT {name: successors.name, modelKey: successors.modelKey, hops: length(succ_path)}) as succs
RETURN {
  current_model: model.name,
  predecessors: [p in preds WHERE p.name IS NOT NULL | p] ORDER BY p.hops DESC,
  successors: [s in succs WHERE s.name IS NOT NULL | s] ORDER BY s.hops
} as model_flows
```

## Data Loading Process

The `bpmn2neo` loader follows this sequence:

1. **Parse BPMN XML** → Extract nodes and relationships
2. **Cleanup** → Remove existing data for the same modelKey (optional)
3. **Ensure Schema** → Create constraints if they don't exist
4. **Load Nodes** → Batch insert all nodes using MERGE
5. **Load Relationships** → Batch insert all relationships
6. **Verify** → Log statistics and validate loading

---

## Design Principles

1. **Semantic Fidelity**: All BPMN 2.0 semantics are preserved
2. **Graph-Native**: Leverages Neo4j's relationship traversal capabilities
3. **Multi-Tenancy**: Container and model keys enable isolation
4. **Query Performance**: Composite constraints and strategic indexing
5. **Extensibility**: Additional properties can be added without schema changes
6. **BPMN Standard Compliance**: Follows OMG BPMN 2.0 specification

## Key Features (v0.2.4)

### 1. Hierarchical Category System
The category system enables flexible organization of BPMN models:

```
Container
  └─ HAS_CATEGORY → Finance (Category)
       ├─ HAS_SUBCATEGORY → Accounting (Category)
       │    └─ CONTAINS_MODEL → InvoiceApproval (BPMNModel)
       └─ HAS_SUBCATEGORY → Budgeting (Category)
            └─ CONTAINS_MODEL → BudgetPlanning (BPMNModel)
```

**Use Cases:**
- Organize models by business domain, department, or process type
- Support multi-tenant environments with isolated category trees
- Enable drill-down navigation and filtered search
- Maintain hierarchical access control boundaries

### 2. Process Sequencing & Flow Analysis
The `NEXT_PROCESS` relationship chains BPMN models in execution order:

```
OrderReceiving → OrderValidation → OrderProcessing → OrderFulfillment → OrderDelivery
```

**Use Cases:**
- **Process Mining**: Analyze end-to-end workflows spanning multiple models
- **LLM Context**: Build 4-hop predecessor/successor windows for Graph-RAG
- **Impact Analysis**: Trace upstream/downstream dependencies
- **Process Documentation**: Auto-generate process flow diagrams

**Example: 4-Hop Context Window**
```
Model: OrderProcessing
├─ Predecessors (4-hop backward)
│   ├─ OrderValidation (-1 hop)
│   ├─ OrderReceiving (-2 hops)
│   └─ CustomerInquiry (-3 hops)
└─ Successors (4-hop forward)
    ├─ OrderFulfillment (+1 hop)
    ├─ OrderDelivery (+2 hops)
    └─ PostSaleSupport (+3 hops)
```

This context enables LLMs to understand where a process fits in the broader business workflow, improving answer quality for process analysis queries.
